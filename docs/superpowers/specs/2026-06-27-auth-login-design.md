# DOC-01 注册与登录系统设计规格

## 1. 规格定位

本规格用于指导 Citeox GEO 品牌 AI 优化系统的注册与登录模块开发。它基于：

- `文档00：系统总则与开发原则 — Codex最终版开发提示词.md`
- `文档01：注册与登录系统 — Codex最终版开发提示词.md`
- 用户要求：不推翻已有部署和宣传页，在现有基础上优化完善，优先跑通后端系统闭环。

本模块是 Citeox 的用户入口，目标是让宣传页流量可以安全、顺畅地转化为真实注册用户，并进入后续品牌创建流程。

## 2. 设计目标

### 2.1 用户目标

用户可以完成以下路径：

1. 从宣传页点击注册入口进入 `/register`。
2. 填写账号、邮箱、手机号、密码、行业、验证码等信息。
3. 注册成功后自动登录。
4. 无品牌用户跳转 `/brand/create`。
5. 已有账号用户可以通过 `/login` 登录。
6. 忘记密码用户可以通过邮箱重置密码。

### 2.2 商业目标

注册登录模块必须达到商业 SaaS 入口标准：

- 看起来可信，不像测试页面。
- 表单提示清晰，用户知道怎么填。
- 错误提示中文、具体、可操作。
- 密码、邮箱、手机号安全存储。
- 能防止恶意刷注册、刷验证码、撞密码。
- 后续可以自然衔接 DOC-02 品牌创建、DOC-03 账户套餐、DOC-04 数据总览。

### 2.3 技术目标

本模块必须提供：

- 前端注册、登录、忘记密码、重置密码页面。
- 后端认证接口。
- 数据库存储结构。
- Redis 验证码与限流。
- 邮件验证码与重置密码邮件。
- JWT 登录状态。
- 审计日志。

## 3. 范围边界

### 3.1 本阶段必须完成

- `/register` 注册页。
- `/login` 登录页。
- `/forgot-password` 忘记密码页。
- `/reset-password` 重置密码页。
- `/api/v1/auth/*` 认证接口。
- 用户表、邀请码表、审计日志表。
- 邮箱验证码。
- 登录失败锁定。
- 用户注册后默认免费套餐。
- 注册成功后跳转 `/brand/create`。
- 登录成功后根据 `has_brand` 跳转 `/dashboard` 或 `/brand/create`。

### 3.2 本阶段不主动修改

- 现有宣传页主体内容。
- 宣传页价格展示逻辑。
- DOC-02 之后的品牌创建具体业务页面。
- 支付功能。
- AI 平台调用功能。

如果现有宣传页注册或登录按钮不能进入正确路由，只允许小范围调整按钮链接，不改宣传页内容和视觉结构。

### 3.3 可分批增强

为了降低第一轮交付风险，下列能力可以分批完成，但最终必须满足 DOC-01：

- 500+ 弱密码库。
- 200+ 临时邮箱黑名单。
- 500+ 行业词库。
- 用户名连续重名后的智能推荐。
- 注册草稿自动保存。
- 欢迎邮件。

第一版可以先做到核心可用与安全达标，第二版补齐完整词库和体验增强。

## 4. 页面设计

### 4.1 注册页 `/register`

桌面端采用左右布局：

- 左侧 40%：品牌区，展示 Citeox Logo 和价值主张。
- 右侧 60%：注册表单。

平板和手机端隐藏左侧品牌区，表单居中或全宽展示。

注册字段：

1. 账号名称，必填。
2. 邮箱，必填。
3. 手机号，必填。
4. 密码，必填。
5. 确认密码，必填。
6. 行业，必填。
7. 邀请码，选填。
8. 邮箱验证码，必填。
9. 用户协议与隐私政策勾选，必选。

注册成功后：

- 显示中文成功提示。
- 自动登录。
- 1.5 秒内跳转 `/brand/create`。

### 4.2 登录页 `/login`

登录字段：

1. 账号：支持用户名、邮箱、手机号。
2. 密码。
3. 记住我。
4. 忘记密码入口。

登录成功后：

- 如果用户已有品牌，跳转 `/dashboard`。
- 如果用户没有品牌，跳转 `/brand/create`。

登录失败时：

- 账号密码错误：提示“账号或密码错误，请重新输入”。
- 账号不存在：提示“该账号不存在，请检查输入或前往注册”。
- 连续失败 5 次：锁定 15 分钟。
- 账号停用：提示联系客服。

### 4.3 忘记密码页 `/forgot-password`

用户输入注册邮箱后，系统发送重置链接。

为了防止别人试探邮箱是否存在，无论邮箱是否存在，前端都显示统一提示：

“如果该邮箱已注册，我们会发送密码重置邮件，请注意查收。”

### 4.4 重置密码页 `/reset-password`

页面通过 token 校验链接是否有效。

状态：

- 有效：允许输入新密码和确认密码。
- 无效或过期：提示“重置链接已失效，请重新申请”。
- 成功：提示“密码重置成功”，跳转登录页。

## 5. 前端交互规则

### 5.1 通用规则

- 所有按钮点击后立即置灰并显示处理中状态。
- 所有异步请求必须有 loading。
- 所有错误提示必须中文、具体、可操作。
- 表单输入支持 Tab、Enter、Esc。
- 输入时 300ms 防抖校验。
- 失焦时立即校验。
- 提交失败时滚动到第一个错误字段。
- 手机端不可出现文字重叠、按钮溢出、输入框被遮挡。

### 5.2 注册字段校验

账号名称：

- 2-20 个字符。
- 支持中文、英文、数字、下划线。
- 不能纯数字。
- 不能包含敏感词。
- 后端查重。

邮箱：

- 格式必须正确。
- 自动去除首尾空格。
- 自动转小写。
- 临时邮箱禁止注册。
- 支持常见邮箱拼写纠错。
- 后端查重。

手机号：

- 必填。
- 中国大陆 11 位手机号。
- 前端展示为 `138 1234 5678`。
- 后端按去空格后的手机号查重。

密码：

- 8-32 位。
- 显示强度条。
- 不能是常见弱密码。
- 不能与用户名相同。
- 后端使用 bcrypt 或 argon2id 存储，不存明文。

行业：

- 必填。
- 支持输入和下拉建议。
- 无法命中词库但看起来合理时，可以作为自定义行业保存。

邀请码：

- 选填。
- 填写时必须为 8 位数字。
- 后端验证是否存在、是否有效、是否过期。

邮箱验证码：

- 6 位数字。
- 有效期 5 分钟。
- 60 秒内不能重复发送。
- 最多尝试 5 次。

协议勾选：

- 未勾选时不发送注册请求。
- 页面给出明显提醒。

## 6. 后端接口设计

所有接口使用 `/api/v1/` 前缀，返回统一格式：

成功：

```json
{ "success": true, "data": {}, "message": "操作成功" }
```

失败：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "邮箱格式不正确，请输入有效的邮箱地址"
  }
}
```

### 6.1 接口清单

- `POST /api/v1/auth/check-username`：检查用户名。
- `POST /api/v1/auth/send-verify-code`：发送邮箱验证码。
- `POST /api/v1/auth/validate-invite-code`：验证邀请码。
- `POST /api/v1/auth/validate-industry`：验证行业。
- `POST /api/v1/auth/register`：注册。
- `POST /api/v1/auth/login`：登录。
- `POST /api/v1/auth/logout`：退出登录。
- `POST /api/v1/auth/refresh`：刷新登录状态。
- `POST /api/v1/auth/forgot-password`：发送重置密码邮件。
- `POST /api/v1/auth/reset-password`：重置密码。
- `GET /api/v1/auth/me`：获取当前用户。

### 6.2 注册接口要求

注册接口必须执行后端二次校验：

- 用户名规则。
- 邮箱格式、黑名单、查重。
- 手机号格式、查重。
- 密码强度与弱密码拦截。
- 行业有效性。
- 邀请码有效性。
- 邮箱验证码有效性。
- 协议同意状态。

注册成功后必须同时完成：

- 创建用户。
- 创建默认免费套餐。
- 创建 API 余额账户，初始余额 0。
- 写入审计日志。
- 设置 httpOnly 登录 Cookie。
- 返回脱敏后的用户信息。

### 6.3 登录接口要求

登录接口支持用户名、邮箱、手机号。

登录成功后：

- 更新最后登录时间。
- 更新最后登录 IP。
- 写入审计日志。
- 设置 access token 和 refresh token。
- 返回 `has_brand`，方便前端跳转。

登录失败后：

- 不返回密码具体错误原因。
- 累计失败次数。
- 连续失败 5 次锁定 15 分钟。

### 6.4 忘记密码接口要求

接口不暴露邮箱是否存在。

如果邮箱存在：

- 生成 30 分钟有效 token。
- 写入 Redis。
- 发送重置邮件。

如果邮箱不存在：

- 仍返回统一成功提示。
- 后端记录一次安全日志。

## 7. 数据库设计

### 7.1 用户表 `users`

核心字段：

- `id`：UUID。
- `username`：账号名称，唯一。
- `email`：加密后的邮箱。
- `email_hash`：邮箱哈希，用于查重和登录。
- `phone`：加密后的手机号。
- `phone_hash`：手机号哈希，用于查重和登录。
- `password_hash`：密码哈希。
- `industry`：行业。
- `plan`：默认 `free`。
- `api_balance`：默认 0。
- `status`：`active`、`suspended`、`deleted`。
- `has_brand`：默认 false。
- `last_login_at`：最后登录时间。
- `last_login_ip`：最后登录 IP。
- `created_at`、`updated_at`。

### 7.2 邀请码表 `invite_codes`

核心字段：

- `id`。
- `code`。
- `max_uses`。
- `used_count`。
- `benefit`。
- `expires_at`。
- `is_active`。

### 7.3 审计日志表 `audit_logs`

记录：

- 注册。
- 登录。
- 退出。
- 密码重置。
- 验证码发送。
- 登录失败锁定。

日志不得保存明文密码、验证码全文、密钥、完整 token。

## 8. Redis 设计

Redis 用于短期状态：

- `email_verify:{email_hash}`：邮箱验证码，5 分钟。
- `email_verify_attempts:{email_hash}`：验证码尝试次数。
- `email_ratelimit:{email_hash}`：验证码 60 秒发送限制。
- `login_attempts:{identifier_hash}`：登录失败次数。
- `login_lock:{identifier_hash}`：登录锁定状态，15 分钟。
- `reset_token:{user_id}`：重置密码 token，30 分钟。

Redis key 不使用明文邮箱和手机号，统一使用 hash。

## 9. 安全设计

### 9.1 密码

- 不存明文密码。
- 使用 bcrypt saltRounds >= 12 或 argon2id。
- 重置密码后旧 refresh token 失效。

### 9.2 邮箱和手机号

- 数据库存储加密值。
- 查重和登录使用 hash。
- API 返回时只返回脱敏值，例如 `1********78`、`c***@qq.com`。

### 9.3 JWT 与 Cookie

- access token 有效期 2 小时。
- refresh token 有效期 7 天。
- Cookie 设置 `httpOnly`、`secure`、`sameSite=strict`。
- 生产环境仅 HTTPS。

### 9.4 限流

- 注册：每 IP 每分钟 5 次。
- 登录：每 IP 每分钟 10 次，连续失败 5 次锁定。
- 验证码：每邮箱 60 秒 1 次。
- 忘记密码：每邮箱 30 分钟 3 次。

### 9.5 错误信息

不得向用户暴露：

- 数据库连接错误。
- Redis 错误。
- 邮件服务错误细节。
- 加密失败细节。
- token 解析错误细节。

统一转为中文提示，例如：

- “服务繁忙，请稍后重试。”
- “网络连接超时，请检查网络后重试。”
- “验证码已失效，请重新获取。”

## 10. 邮件设计

- `[Citeox] 邮箱验证码`

### 10.2 重置密码邮件

- 有效期 30 分钟。

欢迎邮件可放在第二批增强中，不阻塞第一版注册登录闭环。

## 12. 验收清单

### 12.1 用户侧验收

- 可以打开 `/register`。
- 可以填写完整注册表单。
- 可以收到邮箱验证码。
- 验证码错误时有中文提示。
- 注册成功后自动进入 `/brand/create`。
- 可以打开 `/login`。
- 可以使用用户名登录。
- 可以使用邮箱登录。
- 可以使用手机号登录。
- 登录后根据是否有品牌正确跳转。
- 可以退出登录。
- 退出后不能访问需要登录的页面。
- 可以发送重置密码邮件。
- 可以通过重置链接设置新密码。
- 手机端页面不变形。

### 12.2 安全验收

- 数据库看不到明文密码。
- 数据库看不到明文邮箱。
- 数据库看不到明文手机号。
- 验证码不会返回给前端。
- 连续 5 次登录失败会锁定。
- 未登录访问 `/dashboard` 会被拦截。
- 用户信息接口不会返回其他用户数据。
- 错误信息不暴露技术细节。

### 12.3 商用体验验收

- 所有按钮点击后有 loading。
- 所有错误提示中文、具体、可操作。
- 所有表单有即时校验。
- 注册失败不会清空已填内容。
- 邮箱验证码按钮有 60 秒倒计时。
- 忘记密码不暴露邮箱是否存在。
- 页面没有“开发中”“TODO”“undefined”“null”等字样。

## 13. 用户自测方法

用户不需要懂代码，按下面测试：

1. 打开 `https://citeox.com/register`。
2. 故意输入错误邮箱，看是否提示中文错误。
3. 输入弱密码，看是否提示密码不安全。
4. 点击发送验证码，看是否进入倒计时。
5. 填错验证码，看是否提示重新获取或检查。
6. 完成注册，看是否进入品牌创建页面。
7. 退出登录。
8. 打开 `https://citeox.com/login`，用刚才账号登录。
9. 连续输错 5 次密码，看是否锁定。
10. 测试忘记密码邮件。
11. 用手机打开注册页，看是否变形。

## 14. 常见问题排查

页面打不开：

- 检查 Render Web Service 是否运行。
- 检查域名是否指向正确服务。
- 检查构建是否成功。

验证码收不到：

- 检查邮件服务环境变量是否配置。
- 检查发件域名是否验证。
- 检查垃圾邮件箱。
- 检查 Redis 是否正常。

注册失败：

- 检查数据库连接。
- 检查邮箱或手机号是否重复。
- 检查验证码是否过期。
- 检查后端日志中的中文错误码。

登录失败：

- 检查账号是否存在。
- 检查密码是否正确。
- 检查是否被 15 分钟锁定。
- 检查 Cookie 是否被浏览器拦截。

登录后跳转不对：

- 检查 `has_brand` 字段。
- 没有品牌应跳转 `/brand/create`。
- 已有品牌应跳转 `/dashboard`。

## 15. 实施约束

- 不推翻现有宣传页。
- 不直接把密钥写进代码。
- 不在聊天里暴露数据库密码、邮件密钥、JWT 密钥。
- 不把核心安全逻辑放在前端。
- 不在前端展示算法、计费、权限判断细节。
- 任何新增表或字段必须通过迁移脚本管理。
- 每次上线前必须先运行注册、登录、忘记密码的完整测试。

## 16. 第一版交付定义

第一版完成时，用户必须可以真实完成：

```plaintext
宣传页 → 注册 → 邮箱验证 → 自动登录 → 品牌创建入口
```

以及：

```plaintext
登录 → 判断是否已有品牌 → 进入后台或品牌创建
```

只要这两个路径跑通，就说明 DOC-01 对后续模块已经具备可用基础。
