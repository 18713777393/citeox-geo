# DOC-01 注册与登录系统补全计划（中文版）

> **给后续开发 Agent 的要求：**执行本计划前必须先阅读 `文档00：系统总则与开发原则` 和 `文档01：注册与登录系统`。实现时建议使用 `subagent-driven-development` 或 `executing-plans`，逐任务打勾推进。所有代码变更必须先有测试或明确验收步骤。

**目标：**在保留现有宣传页入口和现有后端基础的前提下，把 Citeox 的注册、登录、找回密码、重置密码、安全校验补齐到 DOC-01 可验收标准。

**重要纠正：**`citeox.com` 打开先看到宣传页是正确的。DOC-01 的问题不是“有宣传页”，而是用户从宣传页点击注册/登录后，后续 `/register`、`/login`、`/forgot-password`、`/reset-password` 的页面、字段、接口、安全、跳转和验收标准还没有完全按文档实现。

**当前技术栈：**Node.js 20、Express、TypeScript、Prisma、PostgreSQL、bcryptjs、JWT、Render、单文件静态前端 `frontend/GEOFlow-Integrated-Final-White.html`。

## 一、必须遵守的开发原则

- 不推翻重来：保留现有数据库、后端 API、宣传页和系统主流程，在现有代码上增量补齐。
- 前后端分离：注册、登录、验证码、密码重置等敏感操作必须有后端二次校验。
- 后端为准：前端可以做体验校验，但安全、权限、查重、加密、限流必须以后端为准。
- 商用级品质：前端不能出现“开发中”“TODO”、技术错误、白屏、无 loading 状态。
- 用户信息安全：邮箱、手机号加密或脱敏；密码必须 bcrypt 哈希；API 不泄露其他用户信息。
- 统一 API 规范：优先使用 `/api/v1/auth/*`。
- 统一错误提示：中文、具体、可操作，不暴露数据库、堆栈、内部服务细节。
- 响应式：桌面、平板、手机三端都要可用。
- 验收优先：完成后必须按 DOC-01 逐项验收，不能只说“看起来可以”。

## 二、当前情况判断

### 已确认正常的部分

- `https://citeox.com/` 能打开宣传页。
- `https://citeox.com/register` 和 `https://citeox.com/login` 返回 200。
- 后端健康检查可用：`https://citeox-api.onrender.com/api/health` 返回 200。
- 本地项目目录是：`C:\Users\路\Desktop\源码`。
- 后端已有 auth 基础文件：
  - `backend/src/routes/auth.ts`
  - `backend/src/services/auth.ts`
  - `backend/src/services/authSecurity.ts`
  - `backend/prisma/schema.prisma`

### 已确认的问题

- `/register` 和 `/login` 当前返回的是同一个大型单页 HTML，只是靠路径或 query 切换状态，不是 DOC-01 中清晰独立的注册/登录/找回/重置流程。
- 线上 `POST /api/v1/auth/check-username` 当前返回 401 `UNAUTHENTICATED`，但 DOC-01 要求注册前可以公开查重。
- 前端注册字段语义不完全正确：有“名字”字段，DOC-01 要的是“账号名称”。
- 手机号在部分界面里曾显示为选填，但 DOC-01 要求手机号必填。
- 注册草稿 key 当前不是 DOC-01 要求的 `citeox_register_draft`。
- 前端仍有 `/api/auth/*` 兼容路径调用，DOC-01 标准是 `/api/v1/auth/*`。
- 行业字段不是完整 Combobox。
- 邀请码缺少 DOC-01 要求的 tooltip 和完整校验反馈。
- 用户名连续重名 3 次后的推荐名称标签需要补齐或验证。
- 忘记密码和重置密码需要支持直接访问 `/forgot-password`、`/reset-password?token=xxx`。
- 需要补测试，否则后面很容易再次出现“看似实现，实际漏要求”。

## 四、推荐开发顺序

### 阶段 1：先修后端公开接口和测试

**目的：**先解决注册页无法公开查重的问题，否则前端体验补再多也会卡住。

**涉及文件：**
- `backend/src/app.ts`
- `backend/src/routes/auth.ts`
- `backend/src/routes/compat.ts`
- `backend/src/services/authSecurity.ts`
- `backend/tests/authSecurity.test.ts`
- `backend/tests/authRoutes.public.test.ts`
- `backend/package.json`

**要做：**
- 新增 auth 安全测试。
- 新增公开 auth 路由测试。
- 确认 `/api/v1/auth/check-username` 不需要登录。
- 确认 `/api/v1/auth/validate-industry`、`/validate-invite-code`、`/send-verify-code`、`/forgot-password` 行为符合 DOC-01。
- 修复线上 401 问题。

**验收命令：**

```bash
cd backend
npm run prisma:validate
npm run build
npm run test:auth-security
npm run test:auth-public
```

**线上验收：**

```bash
curl https://citeox-api.onrender.com/api/health
curl -X POST https://citeox-api.onrender.com/api/v1/auth/check-username -H "Content-Type: application/json" -d "{\"username\":\"doc01_probe_user\"}"
```

预期：第二个接口返回用户名可用性 JSON，不能返回 `UNAUTHENTICATED`。

### 阶段 2：整理宣传页到注册/登录的路由链路

**目的：**保留宣传页，但让用户点击注册/登录后进入 DOC-01 规定的页面状态。

**涉及文件：**
- `frontend/GEOFlow-Integrated-Final-White.html`
- 静态部署或 rewrite 配置（如果当前托管平台需要）

**要做：**
- `https://citeox.com/` 保持宣传页。
- `https://citeox.com/register` 直接显示注册页。
- `https://citeox.com/login` 直接显示登录页。
- `https://citeox.com/forgot-password` 直接显示找回密码页。
- `https://citeox.com/reset-password?token=xxx` 直接显示重置密码页。
- 注册/登录相关 API 调用统一改成 `/api/v1/auth/*`。

**验收方式：**

```bash
curl -I https://citeox.com/register
curl -I https://citeox.com/login
curl -I https://citeox.com/forgot-password
curl -I "https://citeox.com/reset-password?token=test"
```

浏览器验收：四个地址必须显示对应页面，不应该都像同一个混合入口。

### 阶段 3：补完整注册页 9 字段

**目的：**让注册页真正符合 DOC-01。

**涉及文件：**
- `frontend/GEOFlow-Integrated-Final-White.html`
- 必要时拆出独立前端文件或组件，但不能大规模推翻系统。

**必须实现：**
- 账号名称：2-20 字符、中文/英文/数字/下划线、不能纯数字、敏感词、查重、简单度警告、连续重名 3 次推荐名称。
- 邮箱：格式校验、临时邮箱拦截、域名纠错、一键修正、查重。
- 手机号：必填、11 位大陆手机号、自动显示为 `138 1234 5678`。
- 密码：8-32 字符、强度条、显示/隐藏、弱密码拦截。
- 确认密码：实时一致性校验。
- 行业：Combobox，支持输入建议、方向键、Enter、Esc。
- 邀请码：选填、8 位数字、tooltip、有效权益提示。
- 邮箱验证码：6 个独立格子、发送状态、60 秒倒计时。
- 协议勾选：未勾选时 shake 动画，不发送请求。
- 草稿保存：使用 `citeox_register_draft`，每 3 秒保存，注册成功清除。
- 提交按钮：点击后立即置灰并显示 `注册中...`，15 秒超时处理。

**验收标准：**
- 注册流程完整走通。
- 任一字段错误时滚动到第一个错误字段。
- 错误提示中文、具体、可操作。
- 注册成功后 1.5 秒跳转 `/brand/create`。

### 阶段 4：补完整登录、找回密码、重置密码

**涉及文件：**
- `frontend/GEOFlow-Integrated-Final-White.html`
- `backend/src/services/auth.ts`
- `backend/src/routes/auth.ts`

**必须实现：**
- 登录账号字段支持用户名、邮箱、手机号。
- 有内容时显示“一键清空”。
- 密码支持显示/隐藏。
- Enter 提交。
- 记住我。
- 账号密码错误、账号不存在、连续 5 次错误、账号停用都有对应中文 Toast。
- 登录成功后：
  - 有品牌：跳转 `/dashboard`
  - 无品牌：跳转 `/brand/create`
- 找回密码统一成功提示，防邮箱枚举。
- 重置密码 token 30 分钟有效，失效时提示重新申请。
- 重置成功 1 秒后跳转登录。

### 阶段 5：安全、审计、部署验收

**必须验证：**
- 邮箱和手机号在数据库中不是明文。
- 密码是 bcrypt 哈希。
- access token 2 小时。
- refresh token 7 天。
- cookie 是 `httpOnly + secure + sameSite=strict`。
- 注册、登录、退出、密码重置都有审计日志。
- 注册、登录、验证码、重置密码有限流。
- 移动端、平板、桌面端布局不破。

**最终验收清单：**

```text
1. 注册流程完整走通：9字段 + 验证码 + 提交 + 跳转
2. 所有校验规则正确触发：前端即时 + 后端二次
3. 邮箱验证码发送、接收、验证正常
4. 智能域名纠错触发正确
5. 密码强度条动画流畅
6. 登录成功根据 has_brand 正确跳转
7. 连续5次错误锁定15分钟
8. 忘记密码到重置密码流程完整
9. 所有错误提示中文、具体、可操作
10. 移动端三档响应式无破损
11. 按钮防重复提交生效
12. API Rate Limiting 生效
13. 手机号、邮箱加密存储
14. 密码 bcrypt 哈希
```

## 五、不能直接继续开发的情况

- 需要修改 Render 生产环境变量时，必须先确认。
- 会影响已有用户数据的数据库迁移，必须先备份并确认。
- 需要使用真实 Resend 邮件发送生产邮件时，必须确认。
- 需要 DNS、域名、静态托管后台权限时，必须确认。
- 同一个错误连续两轮无法定位时，必须暂停并拿新证据，不允许盲目反复改。

## 六、我建议下一步怎么做

先执行阶段 1：修复后端公开 auth 接口和测试。因为现在 `check-username` 线上返回 401，这会直接阻断 DOC-01 注册页的账号名称查重体验。等后端公开接口稳定后，再补前端注册页和登录页，效率最高，也最不容易返工。
