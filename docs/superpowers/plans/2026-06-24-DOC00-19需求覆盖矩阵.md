# Citeox DOC-00~DOC-19 需求覆盖矩阵

**生成时间：**2026-06-24

**目标：**把用户提供的 20 份开发文档逐项对照当前源码，明确哪些已经具备基础、哪些只是占位、哪些必须优先补齐。本文只做审计与规划，不修改业务代码。

## 状态定义

| 状态 | 含义 |
| --- | --- |
| 已实现 | 核心接口、数据结构、前端流程、权限、错误状态基本符合文档，可进入细节验收 |
| 部分实现 | 有核心骨架或部分功能，但接口/字段/体验/权限/测试未覆盖完整 |
| 占位实现 | 有页面或路由名称，但主要逻辑是模拟、本地状态、兼容接口或简化返回 |
| 未实现 | 当前源码没有对应可用能力 |
| 风险实现 | 已有实现可能破坏商用要求，如前端暴露敏感逻辑、路径不一致、权限不足、编码异常 |

## 关键风险清单

| 风险 | 证据 | 影响 | 建议 |
| --- | --- | --- | --- |
| 前端工程不可持续 | `GEOFlow-Integrated-Final-White.html` 单文件 324KB，包含全部页面和大量全局函数 | 后续继续修改会更乱，难测试、难复用、难做权限 | 新功能迁入 React/TS 模块；旧 HTML 只作为过渡原型 |
| 接口路径不统一 | app 同时挂载 `/api/auth`、`/api/v1/auth`、`/api/projects`、`/api/questions` 等 | 前后端对接混乱，文档验收困难 | 新增统一 `/api/v1/*`，旧路径只做兼容 |
| 核心算法不够壁垒 | `geoWorkflow.ts` 的评分主要基于回答摘要字符串信号 | 无法形成可信评分和行业基准，客户难信服 | 建 GEO Engine v1：证据快照、8 维评分、rubric 版本、confidence |
| 缺少真实编排与队列 | 多数 run/generate 是同步接口，DOC 要求 BullMQ/WebSocket | 长任务会超时，用户看不到进度 | 引入 Job/Task 表、BullMQ worker、WebSocket/SSE 进度 |
| 计费闭环未完全覆盖 | 有 AiUsageLog、Order，但刷新/诊断/生成等未全部强制预估与扣费 | 成本失控，商用风险高 | 所有 AI 调用统一通过 CreditService |
| 中文文案编码风险 | 多个源码输出中出现中文乱码字面量 | 前端/接口可能展示乱码，降低商用品质 | 全仓 UTF-8 检查，修复乱码字符串 |
| 测试覆盖不足 | 后端只有 `test:source-hub` | 每轮修改容易回归 | 先补 Auth、Billing、GEO Engine、Admin、E2E 测试 |
| DOC-19 过早实现风险 | 底层 DOC-04~17 多数未稳定 | 自动化会放大错误 | DOC-19 必须最后做 |

## 下一步执行建议

### 第一张开发卡：DOC-01 注册登录验收修复

**目标：**让线上用户从宣传页进入注册/登录，注册流程完全符合 DOC-01，并且注册后进入品牌创建。

**必须验证：**

- `/register`、`/login`、`/forgot-password`、`/reset-password` 路由可直接访问。
- `/api/v1/auth/check-username` 未登录可用。
- 9 个注册字段完整，手机号按 DOC 必填。
- 邮箱验证码、倒计时、验证码失效、重复发送限流可用。
- 密码强度、确认密码、协议勾选、localStorage 草稿可用。
- 注册成功创建默认免费套餐和 ¥0 API 余额。
- 登录、刷新、退出、找回密码链路可用。
- 所有错误中文、具体、无乱码。

### 第二张开发卡：DOC-02 品牌创建向导

**目标：**注册后用户必须创建品牌，品牌创建触发首次诊断任务。

**必须验证：**

- `/brand/create` 三步向导。
- 行业/品类级联、品牌名、网址、目标、平台、竞品、关键词。
- 品牌数量限制、套餐限制、余额检查、预估费用。
- 二次确认与进度弹窗。
- 创建成功后进入数据总览，并能看到采集/诊断任务状态。

### 第三张开发卡：GEO Engine v1 设计

**目标：**先把核心算法边界定住，再实现 DOC-07/08/09/10。

**建议新增概念：**

- `evidence_snapshots`
- `answer_signals`
- `score_rubrics`
- `industry_benchmarks`
- `score_runs`
- `diagnosis_runs`
- `strategy_runs`

**原则：**算法细节只在后端；前端仅展示分值、等级、评语、建议、证据摘要。
