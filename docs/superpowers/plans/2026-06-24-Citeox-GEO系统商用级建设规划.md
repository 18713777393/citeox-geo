# Citeox GEO 系统商用级建设规划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans when turning this roadmap into implementation tasks. Do not implement several DOC modules in one pass.

**目标：**把 Citeox 从“单页原型 + 局部后端能力”升级为可注册、可建品牌、可真实监测、可评分、可诊断、可生成策略、可生产内容、可分发、可复盘、可自动化的商用 GEO SaaS。

**架构原则：**保留现有 Express/Prisma/PostgreSQL/Redis/Render 基础，按 DOC-00 的“不推翻重来”原则做增量治理；但必须把巨大前端 HTML、兼容占位接口、算法散落逻辑逐步迁移为清晰模块。核心算法只在后端运行，前端只展示结论和可操作建议。

**技术栈：**React 18 + TypeScript strict + Tailwind + Framer Motion；Node.js 20 + Express/Nest 风格服务层 + Prisma + PostgreSQL 16 + Redis 7 + BullMQ + WebSocket；Render 部署；Resend 邮件；支付与 AI 平台密钥仅后端环境变量。

## 二、核心判断

1. 你的文档已经形成完整商业闭环：宣传页 → 注册登录 → 创建品牌 → 真实 AI 平台采集 → 问题库 → 回答监控 → 8 维曝光评分 → 差距诊断 → 内容策略 → 素材库 → 内容工厂 → 分发 → 复盘 → 自动优化 → 报告。
2. 当前实现不应继续靠零散对话修补。每次只能锁定一个 DOC 模块，用“需求覆盖矩阵 + 测试 + 页面验收 + API 验收”推进。
3. 技术壁垒不在“生成几篇文章”，而在后端 GEO 引擎：可复验采集证据、版本化评分规则、行业基准、平台差异策略、效果闭环回归。
4. GEOFlow 可以作为内容落地中心参考，尤其是素材、任务、审核、分发、SEO/llms.txt、多站点输出；但它是 PHP/Laravel 系统，不能直接混入当前 Node/React 架构，应按 Apache-2.0 合规参考/迁移思想。
5. 现有后端 `geoWorkflow.ts` 已有问题生成、监控、评分、诊断、策略、内容、报告的雏形，但当前评分仍偏规则/字符串启发式，不足以支撑“核心算法壁垒”。

## 三、必须补强的商用级缺口

### 1. 产品与体验

- 宣传页必须作为第一入口存在，用户被吸引后进入 `/register`，注册后强制进入 `/brand/create`。
- App 内部不应继续以一个巨大 HTML 文件承载全部模块，应逐步拆成路由、组件、服务、状态层。
- 所有模块必须具备 loading、empty、error、retry、disabled、permission、upgrade、mobile 状态。
- 点击生硬的问题要按 DOC-00 改：200ms 内视觉反馈，150-300ms 过渡，异步操作显示进度。

### 2. 注册与品牌创建

- DOC-01 是第一硬门槛：注册 9 字段、验证码、查重、邮箱/手机号加密、bcrypt、限流、审计日志、找回密码必须完整。
- DOC-02 是第二硬门槛：新用户注册后必须完成品牌创建；品牌核心字段提交后锁定，管理员才可审核修改。
- 创建品牌时必须检查套餐、品牌数量限制、API 余额、预估费用，并生成诊断任务。

### 3. GEO 核心算法

后端需要独立 `geo-engine` 能力，不要把评分逻辑混在页面、兼容接口或单个超大 service 里。

建议拆分为：

- `QuestionEngine`：基于品牌、行业、竞品、关键词、平台生成/导入问题。
- `MonitorEngine`：对各 AI 平台执行真实采集，保存原始回答、平台、模型、问题、时间、引用来源。
- `SignalExtractor`：从回答中抽取品牌提及、推荐倾向、引用来源、情感、竞品压制、回答准确性、平台覆盖。
- `ExposureScoreEngine`：8 维评分、行业基准、时间衰减、版本化权重、置信度。
- `GapDiagnosisEngine`：行业基准差距、严重度、拖累评分、修复难度、预计提升。
- `StrategyEngine`：差距 → 问题 → 平台 → 策略模板 → 优先级 → 预计效果。
- `ContentEngine`：策略 + 素材 + 品牌信息 + 平台特征 → 审核前内容草稿。
- `ReviewEngine`：分发前人工审核、风险标签、版本记录。
- `OptimizationEngine`：复盘数据触发下一轮策略。

### 4. 评分可信度

每次评分必须存储：

- `rubric_version`：评分规则版本。
- `evidence_snapshot_id`：本次采集证据集合。
- `platform`、`question_id`、`answer_raw`、`answer_summary`、`sources`、`collected_at`。
- 8 维分数、综合分、等级、评语、变化值。
- `confidence`：样本量、平台失败、回答缺失时降低置信度。

前端仍然不能显示权重和公式，但后台超级管理员可配置算法参数，且要有审计日志。

### 5. 平台化策略

策略不能只写“生成文章”。每条策略至少要落到：

- 目标平台：DeepSeek、豆包、Kimi、通义、文心、元宝、秘塔等。
- 目标问题：来自 DOC-06 问题库。
- 目标差距：来自 DOC-09。
- 内容资产类型：FAQ、对比文、教程、案例、技术说明、品牌介绍、llms.txt、结构化知识页。
- 预期提升维度：品牌出现率、推荐率、引用覆盖、竞品压制、问题覆盖等。
- 执行后复测计划：分发后多久复测、用哪些问题复测、怎样判断有效。

### 6. 计费与后台

- 所有 AI 调用必须先估价、余额校验、原子扣费、失败回滚或补偿。
- 管理后台必须有 AI 平台余额、成本、用户消费、平台毛利、模型消费分布。
- `/api/v1/admin/*` 必须全部 `requireAdmin` + RBAC + 审计日志。
- 支付回调必须具备验签、幂等、金额一致性校验，失败拒绝处理。

## 四、推荐实施顺序

### 第 0 阶段：冻结与盘点

- 备份当前源码、数据库 schema、Render 配置、环境变量清单。
- 建立 DOC-00~DOC-19 需求覆盖矩阵。
- 标记现有实现：已实现、部分实现、占位实现、未实现、风险实现。
- 明确生产域名 `citeox.com`、API 域名、Render 服务、数据库连接、Redis、邮件、支付、AI 平台密钥状态。

### 第 1 阶段：入口闭环

- 完成宣传页 → 注册 → 登录 → 找回密码 → 创建品牌 → 数据总览空状态/加载状态。
- 优先修复 `/api/v1/auth/check-username` 不能要求登录的问题。
- 注册完成后必须进入品牌创建，不允许直接看到空壳后台。

### 第 2 阶段：账户、套餐、余额

- 实现 DOC-03 API 额度账户、充值、套餐权限、用量限制。
- 所有后续 AI 调用必须通过统一 CreditService。
- 后台补充平台成本与用户消费。

### 第 3 阶段：品牌与问题库

- 完成 DOC-02、DOC-05、DOC-06。
- 品牌信息、竞品、关键词、监测平台成为后续算法的稳定输入。
- 问题库支持系统采集、手动导入、去重、热度、覆盖状态。

### 第 4 阶段：GEO 引擎 v1

- 完成 DOC-07、DOC-08、DOC-09。
- 先不要追求复杂 AI 算法，先做到证据链完整、8 维评分稳定、结果可复验。
- 每个评分必须可追溯到回答快照，但前端不暴露算法。

### 第 5 阶段：策略与内容落地

- 完成 DOC-10、DOC-11、DOC-12、DOC-13、DOC-14。
- GEOFlow 只作为内容工程参考：素材、RAG、任务、审核、分发、SEO/llms.txt。
- 所有内容先进入审核，不直接发布。

### 第 6 阶段：复盘、优化、报告

- 完成 DOC-15、DOC-16、DOC-17。
- 把“策略是否有效”接回评分与问题库。
- 报告要能输出 PDF、分享链接、邮件发送。

### 第 7 阶段：一键自动化

- DOC-19 最后做，因为它只是编排层。
- 自动化只能调用已稳定的 DOC-04~DOC-17 API，不能重复写底层逻辑。
- 需要 WebSocket 进度、失败重试、余额不足暂停、执行日志、通知。

## 五、每个模块开发必须执行的流程

1. 读取 DOC-00 和当前模块 DOC。
2. 建立模块需求覆盖矩阵。
3. 对照现有源码，标记复用/修改/新增/删除。
4. 先写后端契约：API、schema、权限、错误码、审计日志。
5. 再写前端状态：loading、empty、error、permission、upgrade、mobile。
6. 加测试：服务层单测、接口测试、关键 E2E。
7. 本地运行并截图/日志验收。
8. 更新覆盖矩阵，不允许口头说“完成”。

## 六、近期最小可执行目标

先做这 3 件事：

1. 建立 `DOC-00~DOC-19 需求覆盖矩阵`，把当前实现逐项打分。
2. 先完成 DOC-01 + DOC-02，因为它们决定用户能否从宣传页进入真正系统。
3. 同步设计 GEO 引擎 v1 的数据库与服务边界，避免后续评分/诊断/策略继续堆在一个文件里。

## 七、明确不建议

- 不建议继续在一个巨大 HTML 文件里添加模块。
- 不建议先做 DOC-19 一键自动化，因为底层模块未稳定时自动化只会放大错误。
- 不建议把 GEOFlow 直接拷进项目运行，技术栈不一致，后期维护成本高。
- 不建议在前端实现任何评分、权重、策略匹配逻辑。
- 不建议先追求“炫酷界面”，应先保证注册、品牌、采集、评分、策略闭环真实可用。
